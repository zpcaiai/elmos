# Skill 1305: b37-extension-marketplace-factory

## Use this skill when

- A repository needs a complete third-party extension platform rather than one ad hoc plugin.
- Language framework recipe comparator runner vertical and policy extensions must share one governed lifecycle.

## Domain-specific risks and invariants

- Untrusted extensions can execute inside customer data boundaries, corrupt migration evidence, or become a software-supply-chain channel.
- Marketplace growth must never bypass tenant isolation, deterministic contracts, certification, or revocation.

## Workflow

1. Inspect Batch 21-36 contracts, extension points, runner boundaries, evidence model, tenant policy, billing, and release infrastructure; create an extension-platform gap inventory.
2. Confirm product, SDK, security, marketplace, finance, legal, support, and customer-data owners plus exact supported product and protocol versions.
3. Scaffold a Marketplace Pack and lock ABI, permission, sandbox, signing, publisher, certification, release, installation, revocation, and commercial policies.
4. Implement one minimal production-shaped extension from local SDK development through sandbox test, signing, publication, installation, execution, evidence, upgrade, rollback, revocation, continuity, recertification, support, settlement, and end-of-life.
5. Run development and negative corpora, then untouched holdout and representative extension workflows; run both the core Batch 37 gate and the complete closure gate.

## Required repository outputs

- `marketplace-packs/<pack-key>/pack.json and support-matrix.json`
- typed SDK and ABI contracts, extension manifests, sandbox and publisher policies
- signed release, dependency lock, runtime health, catalog, publisher lifecycle, recertification, migration, continuity, legal/support, private/offline marketplace, operations, settlement, EOL, and evidence records
- independent development, negative, holdout, and representative-extension corpora

## Verification

- Run all Batch 37 schema, pack, manifest, sandbox, release-chain, and gate validators.
- Build a real extension with the SDK and execute it in the approved sandbox without hidden privileges.
- Verify installation, upgrade, rollback, disable, and revocation against immutable release digests.
- Resolve every evidence reference to an immutable file or approved external reference.

## Stop and escalate when

- Exact product versions, ABI, owner, tenant/data boundary, publisher, or commercial scope is missing.
- The extension requires undeclared filesystem, network, secret, model, repository-write, or process privileges.
- Certification relies only on mocks where real packaging, sandbox execution, signing, or installation is required.

## Definition of done

- One exact extension completes the full governed lifecycle without direct core-code modification.
- Negative tests prove privilege escalation, unsigned release, tenant escape, evidence forgery, and revoked-version execution are blocked.
- Holdout and representative extensions meet the approved profile with no unresolved P0 unknowns.
- The conservative gate emits only the strongest evidence-supported status.
